New extraction and validation pipeline

Each import session moves through explicit steps. The UI reads the same pipeline state the backend stores.

1. ExtractParse CSV, XLSX, XLS, PDF, or pasted table into staging rows.
2. ProfileMeasure column fill rate, samples, empty cells, and likely data kind.
3. AI MapUse Gemini Pro for column mapping, questions, warnings, and payment value guesses.
4. ValidateRun deterministic checks for required fields, duplicates, seats, shifts, allocations, and payments.
5. ReviewGroup attention, collect answers, and let the operator edit guessed row data.
6. CommitCreate students, seats, shifts, allocations, and payments through existing services.

Why split models?

Import onboarding has messy business context and high cost of wrong guesses. Flash is still right for cheaper, simpler interactions.

Pro: schema reasoning Pro: ambiguous column mapping Pro: payment meaning guesses Flash: normal reports/messages unless overridden

Reliability upgrades

The new system treats AI as a helpful guesser, while deterministic services decide what can actually be imported.

AI does not mutate records

AI only returns mappings, questions, warnings, and unconfirmed suggestions.

Manual edits are preserved

Corrected rows are marked as manual normalized data and survive revalidation.

Mapping changes re-normalize rows

Old cached guesses no longer hide new mapping decisions.

Chunked row creation

Large spreadsheets insert staging rows in chunks instead of one giant nested create.

Existing services commit data

Students, seats, shifts, allocations, and payments still go through the domain services.
